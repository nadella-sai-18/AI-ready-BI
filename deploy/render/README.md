# Deploying Metabase + MinusX on Render

Runs **Metabase** and **MinusX OSS** on Render, each on its own automatic HTTPS URL,
both reading your **existing Railway PostgreSQL** as a data source.

**Your Railway frontend / backend / database are NOT touched.** This only adds the two BI
tools on Render. Your database schema stays exactly as it is.

```
Render (auto HTTPS)
  ├─ metabase.onrender.com  ──▶ Metabase ──(settings)──▶ Render Postgres (metabase-metadata)
  └─ minusx.onrender.com    ──▶ MinusX   ──(own state)──▶ pglite volume
                                   │
              both add as a DATA SOURCE ──▶ Railway Postgres (your college data)
```

---

## Can these be Web Services or Docker services on Render?
Both are deployed as **Render Web Services from an existing public Docker image**
(`runtime: image`). **No Dockerfile is needed** — you point Render at the published images:
- Metabase → `docker.io/metabase/metabase:v0.50.21`
- MinusX  → `ghcr.io/minusxai/minusx-frontend:latest`

Render provides the reverse proxy + SSL automatically, so there's **no Nginx or certbot**.

## ⚠️ Read first — Render free-tier limits (so nothing surprises you)
- **Metabase needs memory.** Free instances are 512MB; Metabase is happiest with ≥1GB.
  We cap it with `JAVA_TOOL_OPTIONS=-Xmx400m` so it *fits*, but it may be slow or restart.
  If it keeps crashing, bump the Metabase service to a bigger plan and raise `-Xmx` to `1g`.
- **Free services spin down** after ~15 min idle → the next visit has a slow cold start
  (Metabase can take ~30–60s to wake). Normal on free.
- **Free = no persistent disk**, so MinusX's login/settings reset on each redeploy. To keep
  them, upgrade MinusX to a paid plan and add a disk (the commented `disk:` block in render.yaml).
- **Free Render Postgres expires after ~90 days.** Fine for a project/demo.

These are Render limits, not bugs. Everything still works for a demo.

# PART 1 — Deploy Metabase

### Option A (recommended) — Blueprint (does most of it for you)
1. Push this repo to GitHub (already done).
2. Render Dashboard → **New → Blueprint** → connect this repo → it reads `deploy/render/render.yaml`.
3. Render creates: **metabase-metadata** (Postgres), **metabase**, and **minusx**.
4. It will ask for the values marked `sync: false`:
   - **metabase → `MB_SITE_URL`** — you won't know the URL yet; enter a placeholder like
     `https://metabase.onrender.com`, deploy once, then come back and set it to the **real**
     URL Render gives you (Metabase service → Settings → copy the `.onrender.com` URL) and redeploy.
5. Wait for **metabase** to go **Live** (first boot ~2 min).

### Option B — Manual (if you prefer clicking)
1. Render → **New → PostgreSQL** → name `metabase-metadata`, plan Free → create.
2. Render → **New → Web Service** → **Existing image** → `docker.io/metabase/metabase:v0.50.21`.
3. Set **Environment** variables from `deploy/render/metabase/.env.example` (the `MB_DB_*`
   values come from your new Postgres → *Connections → Internal*). Set `MB_JETTY_PORT=10000`.
4. Create → wait for Live → copy its URL → set `MB_SITE_URL` to that URL → redeploy.

### Connect your COLLEGE data (once Metabase is up)
Metabase → **Admin settings → Databases → Add database → PostgreSQL**:
| Field | Value |
|-------|-------|
| Host | `crossover.proxy.rlwy.net` |
| Port | `32097` |
| Database name | `railway` |
| Username | `postgres` |
| Password | *(from your Railway DB URL)* |
| Use SSL | **ON** |

Save → Metabase syncs your `students`, `marks`, and all `v_*` analytics views.

# PART 2 — Deploy MinusX

If you used the **Blueprint**, MinusX is already created — you just set its secrets:
1. Render → **minusx** service → **Environment**:
   - `ANTHROPIC_API_KEY` = your Claude key (from console.anthropic.com).
   - `NEXTAUTH_SECRET` = already generated by Render ✅.
   - `NEXTAUTH_URL` = deploy once, copy MinusX's `.onrender.com` URL, paste it here, redeploy.
2. Wait for **Live**.

Manual path: **New → Web Service → Existing image** → `ghcr.io/minusxai/minusx-frontend:latest`,
then set the variables from `deploy/render/minusx/.env.example` (remember `PORT=3000`).

### Connect your COLLEGE data (once MinusX is up)
Open MinusX → create the admin workspace → add a **PostgreSQL** data source →
**Connection String** tab:
```
postgresql://postgres:PASS@crossover.proxy.rlwy.net:32097/railway?sslmode=disable
```
Then ask e.g. *"From v_at_risk_students, list the at-risk students"*.

## Environment variables you must fill manually in Render
| Service | Variable | Value |
|---|---|---|
| metabase | `MB_SITE_URL` | your Metabase `.onrender.com` URL (after 1st deploy) |
| minusx | `ANTHROPIC_API_KEY` | your Claude API key |
| minusx | `NEXTAUTH_URL` | your MinusX `.onrender.com` URL (after 1st deploy) |
Everything else (`MB_DB_*`, `NEXTAUTH_SECRET`, ports, DB_TYPE) is set by `render.yaml`.

## Deployment review (issues checked)
- **Port binding:** Metabase listens on `MB_JETTY_PORT=10000` (Render's default routed port);
  MinusX listens on `3000` and we set `PORT=3000` so Render routes there. No conflict — they're
  separate services on separate URLs.
- **Public URL / callbacks:** `MB_SITE_URL` and MinusX `NEXTAUTH_URL` are set to the real
  `https://*.onrender.com` URLs so login redirects / callbacks resolve correctly.
- **CORS:** not applicable — Metabase and MinusX are full apps on their own domains, not called
  cross-origin by your Railway frontend.
- **Database connection:** both reach Railway Postgres via its **public** URL (works from Render).
  Metabase uses **SSL: ON**; MinusX uses `?sslmode=disable`.
- **Internal URLs between MinusX & Metabase:** none required — MinusX queries the **database**
  directly, not Metabase. They are independent.
- **Schema safety:** nothing writes to your college DB except normal read queries; Metabase's own
  tables go into the separate `metabase-metadata` Postgres.
